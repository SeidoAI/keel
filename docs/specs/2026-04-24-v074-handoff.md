## 2. Items

### A. Per-session project-tracking worktrees

**Problem.** `tripwire session spawn` creates one worktree per `session.repos` entry — which today only ever lists code repos (e.g. `SeidoAI/tripwire`). The project-tracking repo (`project-tripwire-ui-init`) is *shared* across all running sessions: every agent writes its `sessions/<id>/...` artifacts and `issues/<KEY>/developer.md` directly into the same on-disk directory. With sequential sessions this is fine. With **parallel** sessions this races on per-issue file writes (rare collision today but inevitable as parallelism increases) and prevents the dual-PR flow from cleanly attributing project-tracking changes to the session that produced them.

**Solution.** Implicitly create a project-tracking worktree per session at prep time, without requiring the user to add it to `session.repos`. Convention over config: the project-tracking repo is `project_dir` itself; its worktree path is `<project_dir>/.git/worktrees/tripwire-wt-<session-id>`; its branch is `proj/<session-id>` (NOT `feat/<session-id>` — distinct prefix so merge-state visually separates project-tracking branches from code-feature branches).

**Prerequisite (one-time, not a code change).** The project-tracking repo needs a git remote. Today `git remote -v` in `project-tripwire-ui-init` returns empty; without a remote, `git push` and `gh pr create` can't run. Whoever runs v0.7.4 setup adds a remote (e.g. `git remote add origin git@github.com:SeidoAI/project-tripwire-ui-init.git` after creating the repo on GitHub). Document this in `WORKFLOWS_INITIAL_SCOPING.md` as a one-line step in the project-bootstrap procedure.

**Code changes:**

1. **`src/tripwire/runtimes/prep.py`** — extend `run()` (around line 411, after `resolve_worktrees` on `session.repos`):
   - Detect if `project_dir` is a git repo with at least one remote: `subprocess.run(["git", "-C", str(project_dir), "remote"], check=False)`. If yes, treat as a project-tracking repo to worktree.
   - Compute `proj_branch = f"proj/{session.id}"`. Compute `proj_wt_path = worktree_path_for_session(project_dir, session.id)` (reuse the existing helper).
   - Call `worktree_add(project_dir, proj_wt_path, proj_branch, "HEAD")` (reuse).
   - Build a `WorktreeEntry(repo="<project-slug>", clone_path=str(project_dir), worktree_path=str(proj_wt_path), branch=proj_branch)` and append to the `worktrees` list passed to `render_claude_md` and stored on `runtime_state`.
   - The agent's CLAUDE.md will then list both worktrees (code + project-tracking). The agent writes session artifacts, developer.md, etc. into the **project-tracking worktree's path** (CLAUDE.md should be updated to say so explicitly — see template change below).

2. **`src/tripwire/templates/worktree/CLAUDE.md.j2`** — update the line currently saying *"Session artifacts (plan.md, verification-checklist.md, issue specs, task-checklist.md) live in the **project-tracking worktree** — the second entry if present"*. This is already accurate; just verify it points to the new project-tracking-worktree path (the second `WorktreeEntry`). No template change needed if path resolution is correct.

3. **`src/tripwire/core/session_pr_flow.py`** — verify the existing dual-PR flow already iterates over every `runtime_state.worktrees` entry (it should per the v0.7.2 spec). If yes, no change needed: it'll auto-push the project-tracking worktree's branch + open a PR for it on `tripwire session complete`. If no, extend it.

4. **`src/tripwire/core/handoff_store.save_handoff`** — handoff.yaml's `branch` field today holds the code-repo branch. With per-session project-tracking worktrees, the handoff needs to either (a) store both branches or (b) keep code-branch as the canonical and let the project-tracking branch be derived from `proj/<session-id>`. **Recommend (b)** — fewer schema changes, and the project-tracking branch is mechanically derivable.

**Cleanup.** `tripwire session cleanup <id>` (already filesystem-aware per PR #19's I6) needs to remove BOTH the code worktree and the project-tracking worktree. Should already work because both worktrees now appear in `runtime_state.worktrees`; verify via test.

**Tests:**
- `tests/unit/test_prep_project_tracking_worktree.py` — NEW. Two cases: (1) project_dir has a remote → project-tracking worktree created with `proj/<sid>` branch + appears in `worktrees` list; (2) project_dir has NO remote → no project-tracking worktree, no error (graceful skip with log line).
- Extend `tests/unit/test_session_cleanup_orphans.py` to confirm both worktrees get removed on cleanup.

**Migration.** Existing in-flight sessions (frontend-views-core, frontend-views-detail launched 2026-04-24) will NOT have project-tracking worktrees because they were spawned pre-v0.7.4. They complete normally on the v0.7.3 path. Only sessions spawned after v0.7.4 land get the new behaviour.

### B. validate-plan path-relativity calibration

**Problem.** The heuristic file-existence check in `tripwire validate-plan` resolves paths relative to the code repo's clone root. Frontend session plans use sub-tree-relative paths (e.g. `src/app/router.tsx`, where the actual file is at `src/tripwire/ui/frontend/src/app/router.tsx`). Result: false-positive `plan/modify_target_missing` warnings on every frontend session.

**Solution — two parts:**

1. **Allow per-session path prefix declaration.** Extend `session.yaml` schema with an optional field `path_prefix: <string>` per repo binding:
   ```yaml
   repos:
     - repo: SeidoAI/tripwire
       base_branch: main
       path_prefix: src/tripwire/ui/frontend
   ```
   When `validate-plan` resolves a `**Files:**` path, it tries each `<clone>/<path>` AND `<clone>/<path_prefix>/<path>` and considers the file existing if either resolves.

2. **Prefix-aware file lookup helper** in `src/tripwire/cli/validate_plan.py`'s `_check_target_existence`. Reuse `session.repos` info to walk both prefix variants per file path. Add a tie-breaker: if both `<clone>/<path>` and `<clone>/<prefix>/<path>` exist, prefer the prefix variant (more specific).

**Schema migration.** Backward-compat: existing sessions without `path_prefix` work unchanged (treated as empty prefix). No required field changes.

**Code changes:**
- `src/tripwire/models/session.py` — extend `RepoBinding` with optional `path_prefix: str | None = None`.
- `src/tripwire/cli/validate_plan.py` — extend `_resolve_repo_paths` to return `list[tuple[Path, str | None]]` (clone + prefix); extend `_check_target_existence` to try both variants.

**Tests:** extend `tests/unit/test_validate_plan_cli.py` with two cases: (1) plan with prefix-relative path resolves correctly when `path_prefix` is set; (2) plan with prefix-relative path produces false-positive warning when `path_prefix` is unset (regression confirmation).

## 7. Risks / open questions for the project agent

- **Project-tracking repo branch convention.** I recommended `proj/<session-id>` to distinguish from `feat/<session-id>`. The project agent should validate this against `branch_naming.derive_branch_name` — that helper currently expects branch types from a fixed enum (feat, fix, refactor, docs, chore, test). Adding `proj` to the default enum is a small change but worth being deliberate about.
- **Initial migration path for existing projects without remotes.** For `project-tripwire-ui-init` specifically, who creates the GitHub repo + adds the remote? The spec says "whoever runs v0.7.4 setup" — needs an owner.
- **`session_pr_flow.py` may already iterate `runtime_state.worktrees`** — if so, A's PR-side change is zero. If not, the project agent needs to extend it. Either way: verify before committing to scope.
- **`path_prefix` schema field name.** Could also be called `path_root`, `subtree`, `relative_to`. Recommend `path_prefix` for clarity but happy to be overridden.
