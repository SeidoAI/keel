# Tripwire v0.7.5 — handoff spec

**Status**: spec (ready for plan)
**Date**: 2026-04-24
**Depends on**: v0.7.4 (`2026-04-24-v074-handoff.md`) — per-session
project-tracking worktrees, validate-plan path-relativity, frontend
copy retunes.
**Sibling**: v0.8 active concept graph
(`2026-04-24-v08-bidirectional-concept-graph.md`) — independent track;
v0.7.5 must not pull v0.8 work forward.
**Source**: cross-cutting analysis of the parallel v0.7.3 trial
(`frontend-views-core` #21 + `frontend-views-detail` #22, 2026-04-24,
~30 min wall clock, $33.82 combined) plus observations about
in-flight-session visibility.

---

## 1. Context

The first multi-session trial of the v0.7.3 spawn template produced
exceptional self-reviews and clean exits — but cross-cutting analysis
surfaced patterns no single-PR review catches:

1. **In-flight visibility is zero.** Until the agent opens the PR at
   complete, the only signal into a live session is the stream-json
   log. No stable URL, no live diff, no continuous CI, no crash
   recovery. Both trial PRs landed one squash commit at the end —
   partly because there's no half-way stable artifact to stage against.
2. **Stop-and-ask is silent on mid-execution scope creep.** Both agents
   made unilateral expansions (TS interface widening, DTO additions,
   library-version substitution, `<Toaster/>` mount). The trigger only
   fires on plan obsolescence, not "I need to expand scope to ship".
3. **Both agents skipped TDD with the same rationalization** ("spec
   was detailed enough that the red step was tautological"). Two
   independent agents reaching the same workaround = pattern, not
   lapse. The skill reference in CLAUDE.md is too easy to treat as
   optional.
4. **Both shipped one squash commit** because the exit protocol's
   "PR-open is your exit signal" implicitly encourages stage-then-PR
   with no incremental checkpoints. Unreviewable diffs at scale.
5. **No formalised PM-review-then-feedback cycle.** PM ad-hoc reviews
   PRs, posts comments, and hopes the agent picks up the feedback
   somehow. No canonical "respawn for fixes" path.
6. **No structured PR-review artifact.** PM observations are buried
   in PR comments. Pattern detection across PRs requires re-reading
   every comment — doesn't compose into data.

v0.7.5 closes all six with five items. The draft-PR-at-session-start
work (A) provides the delivery substrate the review-feedback cycle
(C) lives on top of — they co-design.

## 2. Items

### A. Draft PRs at session-start

**Problem.** Today `session spawn` creates the worktree + starts the
agent but does no remote git action until `session complete`. Result:
no visibility, no continuous CI, no crash recovery. Agents tend to
land one end-of-session commit because there's no live artifact to
push against.

**Solution.** Immediately after each `worktree_add`, emit an empty
marker commit, push, and open a draft PR on the resulting branch.
At `complete`, flip the draft(s) to ready instead of creating new
PRs. Pairs cleanly with v0.7.4's dual-worktree model — each session
opens two draft PRs (code repo + project-tracking repo), each flipped
at complete.

**The wrinkle.** `gh pr create` refuses if there are zero commits
between the feature branch and base. Worktree creation + checkout
is not a commit.

- **Option (a): empty marker commit at prep time.** After
  `worktree_add`, run
  `git commit --allow-empty -m "session(<id>): start"` in the
  worktree, then `git push -u origin <branch>` and
  `gh pr create --draft`. Squash-merge at complete erases the marker
  from history. One line of code, low brittleness. **Recommend (a).**
- **Option (b): deferred-creation on first real commit.** Hook or
  wrapper on the agent's first `git commit` that also pushes + opens
  the draft PR. More involved; unpredictable timing.

**Five wins:**

1. **Visibility from minute one.** Stable URL, live diff, in-browser
   comment threads.
2. **Continuous CI feedback.** Every push triggers tests/lint. Agent
   finds breakage at commit time, not at final-verify.
3. **Crash recovery.** If the process dies mid-session (network, OOM,
   runtime bug), the PR exists with whatever commits already landed.
4. **Clean pairing with v0.7.4.** The dual-PR flow at `complete` flips
   both draft PRs to ready instead of creating them. Same GH-operation
   count, rearranged timing.
5. **Reinforces commit-early discipline** (see item B3 below) — a
   live draft PR nudges toward smaller themed commits because there's
   already something to push to.

**Code changes:**

- `src/tripwire/runtimes/prep.py` — after each `worktree_add` (both
  the code-repo worktree in `resolve_worktrees` and the project-
  tracking worktree in `maybe_add_project_tracking_worktree`), emit
  the empty session-start commit, push, `gh pr create --draft`. Store
  the PR URL on the `WorktreeEntry` (new field `draft_pr_url: str |
  None`).
- `src/tripwire/core/session_complete.py::_flip_drafts_to_ready` (new)
  — replace the pre-v0.7.5 "create PR" step with a "flip to ready"
  step: `gh pr ready <number>` per worktree. Fall back to
  `gh pr create` if `draft_pr_url` is unset (belt-and-braces for
  in-flight sessions started before v0.7.5 lands).
- `src/tripwire/cli/session.py::session_abandon_cmd` — close the draft
  PR(s) via `gh pr close` so they don't pile up as orphan drafts.
- Log the draft PR URL(s) on successful spawn so the operator is one
  click from the live view.

**Tests:**

- `tests/unit/test_prep_draft_pr.py` — NEW:
  - `test_spawn_opens_draft_pr_per_worktree` — subprocess mocks
    capture `gh pr create --draft` calls; assert one per worktree
    (code + project-tracking when applicable) with correct `--base`.
  - `test_spawn_empty_marker_commit_precedes_push` — commit history
    in the worktree has exactly one empty commit titled
    `session(<id>): start` before push.
  - `test_spawn_stores_draft_pr_url_on_worktree_entry` — assert
    `runtime_state.worktrees[i].draft_pr_url` is populated.
  - `test_no_gh_tool_fails_spawn_fast` — when `gh` is missing, spawn
    errors before any mutation happens. Fast-fail beats "8 minutes
    in, complete can't create PR".
- `tests/unit/test_session_complete_flip_draft.py` — NEW:
  - `test_complete_flips_drafts_to_ready` — mocks `gh pr ready`;
    asserts one call per worktree.
  - `test_complete_falls_back_to_pr_create_when_no_draft_url` —
    sessions pre-dating v0.7.5 have `draft_pr_url=None`; complete
    calls `gh pr create` as before.
- `tests/unit/test_session_abandon_closes_drafts.py` — NEW: assert
  `gh pr close` called per worktree when `session abandon` runs.

### B. Spawn template tightening

**File:** `src/tripwire/templates/spawn/defaults.yaml`

Three independent insertions closing three trial-surfaced gaps.

#### B1. Mid-execution scope-creep disclosure

After the headless warning block, before the exit protocol, insert:

```
If during execution you make a change not described in the plan
(new dependency, new DTO field, interface widening, library version
substitution, mounting a globally-scoped UI element, etc.), append
it to `decisions.md` in the project-tracking worktree before
continuing. One paragraph per decision: what you did, what the plan
said vs reality, why your choice was the minimum scope to meet the
stated AC. Do NOT treat scope expansions as implicit — name them.
This file is part of your PR; the PM reviews it alongside the
diff.
```

Converts silent unilateral decisions into a structured reviewable
trail. The existing self-review behaviour already documents these;
making them a separate artifact (a) surfaces them at PR-open time
not post-PR, and (b) makes them grep-able for cross-PR pattern
detection.

#### B2. Explicit TDD mandate

After B1, before the exit protocol, insert:

```
Test-Driven Development is required for every component you write
that has observable behaviour. Sequence per component:
  1. Write a failing test that asserts the AC.
  2. Commit the red test with message `test: <component> red`.
  3. Implement the component until the test passes.
  4. Commit the implementation with message
     `feat(<key>): <component>`.
The `superpowers:test-driven-development` skill defines edge cases.
"The spec was detailed enough that the red step was tautological"
is not a valid skip — write the test anyway. Two parallel sessions
in v0.7.3 used that exact rationalization; we caught it. Don't
repeat it.
```

Naming the prior failure mode in the prompt is a deliberate
anti-rationalization tactic.

#### B3. Per-step commit checkpoints

Modify the existing exit-protocol block (begins "Exit protocol — how
you signal you're done"). Before the existing "PR-open is your exit
signal" paragraph, insert:

```
Commit cadence during execution:
- After each numbered plan step, commit your changes with message
  `step-N(<key>): <step title>` before moving to the next step.
- This breaks the single-squash anti-pattern that makes large PRs
  unreviewable. The PM gets natural rollback points. Squash on
  merge is the PR's job, not yours.
```

Doesn't change the exit signal — changes the commit graph leading
to it. Pairs with A's draft-PR: each step-commit pushes to the live
draft, triggering incremental CI.

### C. Review-feedback cycle

#### C1. Update `resume_prompt_template` in `defaults.yaml`

Today's template assumes the PM updated `plan.md` with a
`## PM follow-up` section. With draft PRs live (item A), PR comments
become the primary feedback channel — point the agent there too:

```yaml
resume_prompt_template: |
  Resuming session {session_id}.

  The PM has provided feedback. Two places to look:
  1. PR comments on your PR — read every PM comment posted since
     your last commit. Each PM comment may include specific bug
     findings with suggested fixes; address each one explicitly.
  2. The plan at {plan_path} — re-read any `## PM follow-up`
     section the PM may have added.

  Your previous conversation is preserved — you have the full
  context of what you've already tried. Continue from where you
  stopped, addressing the PM's feedback in priority order
  (criticals first, then real bugs, then nice-to-fix).

  Exit protocol unchanged: when fixes land, push + the PR auto-
  updates. Post a short follow-up comment on the PR summarising
  what you changed in response to each PM finding. If you disagree
  with a finding, write your reasoning in the follow-up comment
  rather than silently ignoring it.
```

#### C2. Add `tripwire session reopen <session-id>`

Today, completed sessions can't be re-spawned because the status
guard rejects `completed` → `executing`. Add an explicit `reopen`
command (preferred over loosening the guard — keeps the lifecycle
state machine explicit).

```python
@session_cmd.command("reopen")
@click.argument("session_id")
@click.option("--reason", required=True, help="Why are you reopening?")
@click.option("--project-dir", ...)
def session_reopen_cmd(session_id, reason, project_dir):
    """Move a completed session back to executing for PR-fix iteration.

    Records the reopening in `~/.tripwire/logs/<project>/audit.jsonl`
    and appends a `## PM follow-up` section stub (pointing at the PR
    URL) to the session's plan.md if one isn't already present. The
    agent is then re-spawned with `tripwire session spawn <id>
    --resume`.
    """
```

Also flips the draft PR(s) from ready back to draft if they were
already flipped — `gh pr ready --undo` per recorded worktree. Keeps
the "draft while work is happening, ready when merge-ready" contract
consistent.

PM's loop:

```
tripwire session reopen <id> --reason "PR review feedback"
tripwire session spawn <id> --resume
```

Kept as two commands (not folded into `reopen --spawn`) so PM can
update the plan between them if needed.

#### C3. Codify the cycle in `WORKFLOWS_REVIEW.md`

Add a top-level section "Review feedback cycle":

```
1. Agent ships PR. Process exits cleanly per the exit protocol. The
   draft PR (open since spawn-start, item A) flips to ready at
   `session complete`.
2. PM reviews:
   a. Read the agent's self-review (the four-lens block from the
      spawn template).
   b. Walk the verification-checklist independently.
   c. Run `tripwire validate --strict` against the project repo.
   d. Optionally dispatch sub-agents for independent bug scans
      and cross-cutting pattern analysis (see §"Pattern detection"
      below).
3. PM writes the structured PR-review artifact (item D).
4. PM posts findings on the PR — one comment per concern, each
   with a suggested fix. Mark severity (critical / real bug / nice-
   to-fix). The artifact is the PM's view; the PR comments are the
   agent's actionable list.
5. PM either approves+merges (no findings) OR
   `tripwire session reopen <id> --reason "..."` then
   `tripwire session spawn <id> --resume`.
6. Agent (resumed) reads PR comments + plan.md `## PM follow-up`
   section. Addresses each finding. Pushes. Posts a follow-up
   comment summarising what they changed per finding.
7. Loop 2-6 until verdict is approve+merge.
8. `tripwire session complete <id>`. Status → completed.
```

Document in CLAUDE.md too.

### D. Structured `pm-review.md` artifact

#### D1. Location

`sessions/<session-id>/artifacts/pm-review.md`. Lives in the
project-tracking repo as the structured per-iteration record (the
agent's self-review stays inline in the PR).

Multi-cycle sessions append a new top-level section per iteration
(`## Review iteration 1 (...)`, `## Review iteration 2 (...)`). The
frontmatter stays at the top and reflects the **latest** iteration's
verdict; the body carries the historical record.

#### D2. Schema

```yaml
---
session_id: frontend-views-core
pr_number: 21
reviewed_at: 2026-04-24T18:30:00Z
reviewer: pm-agent
iteration: 2
verdict: request_changes      # approved | request_changes | needs_rework | abandoned
recommended_next: respawn_for_fixes  # merge | respawn_for_fixes | abandon
# Per-lens findings count
lens_findings:
  ac_met_but_not_really: 4
  unilateral_decisions: 5
  skipped_workflow: 3
  quality_degradation: 0
# Independent bug scan
bug_findings_critical: 0
bug_findings_real: 2
bug_findings_minor: 2
# Behavioural signals — yes / partial / no / not_triggered
template_adherence:
  reconnaissance_pass: yes
  stop_and_ask_threshold: not_triggered
  four_lens_self_review: yes
  scope_creep_disclosure: no       # missing in v0.7.3; present from v0.7.5
  tdd_followed: no                 # both v0.7.3 sessions skipped
  per_step_commits: no             # both v0.7.3 sessions used 1 squash commit
  exit_on_pr_open: yes
# Cross-PR pattern markers
cross_pr_patterns:
  - "TDD skipped with rationale 'spec was detailed enough'"
  - "1-squash-commit pattern across both parallel sessions"
  - "Unilateral DTO/interface widening to ship feature"
---

## Review iteration 1 (2026-04-24T18:30:00Z) — verdict: request_changes

### Critical
1. ...

### Real bugs
1. ...

### Nice to fix
1. ...

### Cross-PR patterns
- ...

### Recommended next
respawn_for_fixes — see PR comments for the actionable list.
```

#### D3. Why structured

- **Cross-iteration history.** Each review cycle appends; the file
  becomes the audit log of "how many round trips this session took,
  what got caught at which iteration".
- **Cross-session pattern detection.** A future `tripwire pm-review
  patterns` CLI can grep all `pm-review.md` files for fields like
  `tdd_followed=no` and surface aggregates. Without structure you'd
  be diffing prose.
- **Template-iteration feedback loop.** The `template_adherence`
  block tells us over time which prompt-template instructions are
  landing and which are getting skipped. Direct input to v0.8/v0.9
  template tuning.
- **Verdict trend.** A session that takes 4 iterations to merge is a
  different beast from one that takes 1 — the artifact captures it.

#### D4. Manifest update

Add to `src/tripwire/templates/artifacts/manifest.yaml`:

```yaml
- name: pm-review
  file: pm-review.md
  template: pm-review.md.j2
  produced_at: in_review
  produced_by: pm
  owned_by: pm
  required: true
  approval_gate: false
```

`produced_at: in_review` — written when the session is in PM-review
state. Add a corresponding template at
`src/tripwire/templates/artifacts/pm-review.md.j2` rendering the
frontmatter scaffold so `tripwire session scaffold --artifact
pm-review.md` works.

#### D5. CLI helper (v0.7.6 polish)

`tripwire pm-review init <session-id>` — scaffold the next iteration
block in an existing `pm-review.md`, pre-filling `iteration: <N+1>`
and a stub frontmatter. Not needed for v0.7.5 — PM can write the
file directly per the schema. File as v0.7.6 polish if usage shows
it'd help.

### E. Pattern-detection guidance in `WORKFLOWS_REVIEW.md`

Add a new section "Pattern detection across PRs":

```
PM review covers more than per-PR critique. When multiple sessions
have shipped close in time (same week, same template version),
examine PRs together for patterns. Patterns reveal template gaps
that single-PR critique can't see — by definition they're shared
behaviours, not individual lapses.

Patterns to scan for:

| Category | What to look for | Where to record |
|---|---|---|
| Workflow adherence | TDD skipped? Single-squash-commit? Skill never read? | `pm-review.md` `template_adherence` block |
| Decision discipline | Unilateral scope-expansion themes (which kinds recur?) | `cross_pr_patterns` list |
| Common failure modes | State isolation bugs (cache keys, tab state, pagination)? Async ordering? | `bug_findings` + `cross_pr_patterns` |
| Plan-vs-reality | Reconnaissance findings — what kind of obsolescence keeps recurring? | `cross_pr_patterns` |

If a pattern recurs across 2+ sessions, file it as a v0.X.Y template
addition (or a v0.8 spec item) — don't expect agent self-discipline
to fix it. Templates are leverage; reminders aren't.

Use sub-agents (Explore, code-review, general-purpose) for the
independent bug scan and cross-cutting pattern pass. Reading every
diff yourself doesn't scale; sub-agents see what you'd miss when
fatigued.
```

Also extend `WORKFLOWS_REVIEW.md`'s existing red-flags table with
two new rows:

```
| Agent thought | Reality |
| "I'll just review this PR in isolation, the patterns are obvious" | If 2+ sessions completed close in time, scan together. Single-PR review misses discipline patterns that span sessions. |
| "The PR comment captures everything, I don't need pm-review.md" | The PR comment is for the dev agent. pm-review.md is for the data layer that surfaces template-iteration patterns over time. Both are needed. |
```

## 6. Verification

End-to-end after all items land:

1. **Draft PRs visible from minute one.** `session spawn <id>` in a
   repo with a remote: confirm a draft PR exists on each
   worktree's branch with URL logged to the operator. Stream-json
   log prints the URL as one of the first lines.
2. **Complete flips drafts to ready.** `session complete <id>`: `gh
   pr view <number>` reports `isDraft: false`; no new PR was
   created.
3. **Abandon closes drafts.** `session abandon <id>`: `gh pr view
   <number>` reports `state: CLOSED`.
4. **Reopen flips ready → draft.** After complete, run `session
   reopen <id> --reason "..."`; `gh pr view <number>` reports
   `isDraft: true` again.
5. **Resume template cites both PR comments + plan.md.** Post-reopen
   `spawn --resume`: rendered kickoff at
   `<worktree>/.tripwire/kickoff.md` references both "PR comments"
   and "PM follow-up section in plan.md".
6. **Scope-creep disclosure required.** Run a session whose work
   necessarily expands scope; confirm the agent writes
   `decisions.md` in the project-tracking worktree before opening
   the PR. Soft check — prompt asked, agent may deviate but
   `pm-review.md` captures the gap.
7. **TDD mandate enforced.** Inspect commit graph: `test:
   <component> red` commits before `feat: <component>` commits. A
   single squash commit signals B2 didn't land.
8. **Per-step commits.** Commit count on the branch tip ≥ N where N
   = number of plan steps (approximate; squash-on-merge collapses
   for the merged history but the branch has them).
9. **`pm-review.md` scaffold.** `tripwire session scaffold <id>
   --artifact pm-review.md` renders the frontmatter scaffold.
   Validator's `artifact/missing` fires if a session reaches
   `verified` without one (`produced_at: in_review, required:
   true`).
10. **Cross-PR pattern detection workflow.** Run two parallel
    sessions on v0.7.5+, complete both, write `pm-review.md` for
    each. Manual scan surfaces common `template_adherence` values;
    `cross_pr_patterns` captures recurring themes.
11. **Full suite.** `uv run python -m pytest tests/ -q` — ≥1629
    baseline + new tests passing.
12. **Lint.** `uv run ruff check && uv run ruff format --check` —
    clean.
13. **Smoke.** `tripwire --version` → `tripwire, version 0.7.5`.

## 7. Risks / open questions for the project agent

- **`gh` availability at spawn time.** Draft-PR creation requires
  `gh` authenticated with push access to the repo. If missing,
  spawn should fail FAST (before worktree_add) rather than 8
  minutes in at complete. Verify precondition during `prep.run()`.
- **Squash-merge dependency.** The empty `session(<id>): start`
  commit relies on squash-merge to disappear. If a project enforces
  merge-commit, the marker stays in history — tolerable (one noisy
  commit per session) but worth calling out in
  WORKFLOWS_INITIAL_SCOPING.md.
- **Orphan draft PRs across `abandon` failures.** If `session
  abandon` errors between worktree cleanup and PR close, a draft PR
  can leak. Mitigation: a nightly `tripwire drafts reconcile`
  cron-style command (future work) that closes draft PRs whose
  session has status `abandoned` or whose branch no longer exists.
- **CI-minutes cost.** Reconnaissance + early step commits push to
  the draft PR and trigger CI runs that wouldn't otherwise happen.
  Marginal if CI isn't per-minute billed; real if it is. Acceptable
  given benefit 2 (CI-feedback).
- **`decisions.md` location.** Spec says project-tracking worktree.
  Could also live at `sessions/<id>/artifacts/decisions.md` for
  consistency with other artifacts. Sub-question: manifest entry
  (formal) or agent-owned ad-hoc? **Recommend manifest entry at
  `produced_at: implementing, produced_by: execution-agent,
  required: false`** so validators notice but don't block.
- **Iteration counting on `pm-review.md` frontmatter.** Frontmatter
  holds the LATEST iteration's verdict. Alternative: list of
  iteration objects in frontmatter. Recommend sticking with latest-
  only — heavier schema is harder to author by hand.
- **Reopen + resume as one command.** `tripwire session reopen`
  always precedes `session spawn --resume`. Could fold
  (`reopen --spawn`). **Recommend keep separate** — PM needs the gap
  to write the `## PM follow-up` section.
- **Auto-generated `## PM follow-up` stub on reopen.** Append a
  placeholder pointing at the PR URL if one isn't present? Probably
  yes — closes the "PM forgot to update plan" failure mode. One-
  liner in the reopen command.

## 8. Why this matters

Every template tightening up to now has been reactive — wait for an
agent to fail in a new way, then fix the prompt. v0.7.5 makes three
loops explicit:

- **Visibility loop (A).** Operators see live sessions in the
  browser, not in stream-json logs. Crashes lose nothing.
- **Feedback loop (C).** PR → PM review → `session reopen` → agent
  resumes from PR comments. No more "PM commented, hope agent
  picks it up somehow".
