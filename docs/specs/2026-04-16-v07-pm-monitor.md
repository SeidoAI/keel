# Keel v0.7 — PM session lifecycle (monitor + review + complete + spawn refinements)

**Status**: idea (not yet designed)
**Date**: 2026-04-16
**Depends on**: v0.6 session-spawn + worktrees
**Source**: gaps surfaced while launching, reviewing, and completing
parallel execution sessions from a PM agent. Spawn refinements
informed by Claude Code CLI audit (v2.1.110).

---

## 1. The problem

After `/pm-session-spawn` launches two (or more) executing agents
into worktrees, the PM agent goes idle. It has no way to track what
those agents are doing until a PR appears or the human checks in.

The human ends up as the integration layer: tabbing between
terminals, reading logs, relaying status between agents. That's the
PM agent's job.

Additionally, the v0.6 spawn invocation leaves value on the table —
Claude Code exposes flags for effort level, session naming, cost
caps, model selection, streaming output, and session resumption by
name, none of which the v0.6 spawn spec uses. v0.7 corrects this.

## 2. Feature A — session monitor (`/pm-session-monitor`)

### 2.1 Two monitoring modes

The monitor has two data sources, used in priority order:

**Primary: log tailing (stream-json).** When `keel session spawn`
uses `--output-format stream-json` (see §5), the spawn log is a
newline-delimited JSON stream of every Claude Code event: tool
calls, assistant messages, errors, turn counts. The monitor reads
this in real-time via `tail -f <log-path>` — no polling delay, no
git overhead.

**Fallback: git + GitHub polling.** When the log isn't available
(remote sessions, crashed logger), the monitor falls back to:

| Signal | Source | How to read |
|---|---|---|
| New commits | Git branch in worktree | `git -C <wt> log --oneline -5` |
| PR opened | GitHub | `gh pr list --head <branch>` |
| CI status | GitHub | `gh pr checks <pr-number>` |
| CI failure details | GitHub | `gh api repos/.../check-runs` |
| Agent process alive | Session name | `claude --resume <name> --status` or `kill -0 <pid>` |
| PR review comments | GitHub | `gh pr view <number> --comments` |

### 2.2 Actions the PM takes

| Event | PM action |
|---|---|
| Agent committed progress | Log it; update session task-checklist if agent hasn't |
| PR opened | Run `/pm-session-review` on the PR |
| CI failed | Read failure, create a fix issue or send the agent a message |
| Agent process died | Mark session as `failed`, suggest `keel session spawn <id> --resume` |
| Agent stuck (no events for N minutes) | Alert the human; optionally nudge |
| Agent completed (all plan steps done) | Transition session to `completed`, update issue statuses |
| Cost threshold approaching | Alert the human (from stream-json budget events) |
| All sessions completed | Run gap analysis on the next wave, recommend next launches |

### 2.3 Interface

**Slash command:**

```
/pm-session-monitor [session-id ...] [--interval <seconds>]
```

- No args: monitor all `active` sessions.
- With session ids: monitor only those.
- `--interval`: polling interval for fallback mode (default:
  self-paced via `/loop` dynamic mode).

**CLI:**

```
keel session monitor [session-id ...] [--interval <seconds>] [--format text|json]
```

One-shot snapshot. The slash command wraps this in a loop.

### 2.4 Output per tick

```
[14:32] Session monitor — 2 active, 0 completed, 0 failed

  backend-server-scaffold  ● active   turn 23/200  $1.42/$50
    source: stream-json (live)
    latest tool: Edit src/keel/ui/server.py
    branch: feat/backend-server-scaffold (3 commits)
    PR: not yet opened

  frontend-markdown        ● active   turn 8/200   $0.38/$50
    source: stream-json (live)
    latest tool: Write src/components/MarkdownBody.tsx
    branch: feat/frontend-markdown (1 commit)
    PR: not yet opened

  ⚑ No action needed. Next check in ~90s.
```

When something needs attention:

```
[14:47] Session monitor — 1 active, 1 failed

  backend-server-scaffold  ● active   turn 45/200  $3.20/$50
    PR: #4 opened → CI running

  frontend-markdown        ✗ FAILED   turn 18/200  $0.92/$50
    last event: Error — test_MarkdownBody.test.tsx assertion failed
    process: exited
    resumable: yes (claude --resume frontend-markdown)

  ⚑ ACTION: frontend-markdown failed. Options:
    1. Re-engage: keel session spawn frontend-markdown --resume
    2. Alert human
```

### 2.5 Self-pacing logic

**Stream-json mode:** 60-90s between ticks (read latest events from
log, summarise). Cache-friendly interval.

**Polling fallback:**
- Active commits flowing: 60-90s
- Waiting for CI: 120-270s
- Idle (no commits for 10+ min): 600-1200s
- Event detected: immediate action, then resume

## 3. Feature B — session review (`/pm-session-review`)

### 3.1 The gap

The existing `/pm-review` reviews PRs against the **project repo**
(issue/node/session YAML). It does not review **code PRs** opened
by executing agents against the target repo.

When done manually for the keel-ui-init project's first two
sessions, the PM rubber-stamped both without comparing diffs to
issue specs. A proper review done later found: a scoping gap
(no TypeScript types file), premature unspec'd dependencies,
stale path references, a missing component, and layout divergence.

### 3.2 Interface

**Slash command:**

```
/pm-session-review <session-id> [--pr <number>]
```

**CLI:**

```
keel session review <session-id> [--pr <number>] [--format text|json]
```

### 3.3 What the review checks

**Per-issue acceptance criteria verification:**
For each issue in `session.yaml.issues[]`:
1. Read `issues/<key>/issue.yaml` — extract acceptance criteria
2. Read the PR diff — map changed files to the issue's Repo scope
3. For each criterion: evidence in the diff that it's met?
4. Flag unverified criteria

**Deviation detection:**
1. Files touched vs issue Repo scope — flag unspec'd files
2. Dependencies added vs what issues call for — flag extras
3. Directory layout vs Repo scope paths — flag divergence

**Plan adherence:**
1. For each plan.md step: matching files/changes in the diff?
2. Flag steps with no evidence

**Stop-and-ask audit:**
1. Read Execution constraints for "stop and ask" clauses
2. Check if triggered conditions are visible in the diff without
   the agent having stopped

### 3.4 Output

```
PM Session Review: foundation-packaging (PR #2)

Issues: KUI-2, KUI-3, KUI-4, KUI-5, KUI-6
Status: APPROVED with notes

Per-issue:
  KUI-2: 5/5 criteria met
  KUI-3: 4/4 criteria met
  KUI-4: 6/6 criteria met
  KUI-5: 5/5 criteria met — note: issue spec references
         graph/nodes/*.yaml (v0.4 path), agent used nodes/*.yaml
         (correct for v0.5). Issue spec should be updated.
  KUI-6: 5/5 criteria met

Deviations:
  - dependencies.py created (4 lines, stub) — not in any issue scope

Plan adherence: 6/6 steps matched
Stop-and-ask audit: 0 triggered conditions, 0 violations

Recommendation: APPROVE
  Follow-ups:
  - Fix stale path in KUI-5 issue spec
```

## 4. Feature C — session complete (`/pm-session-complete`)

### 4.1 The gap

v0.6 ships create, launch, spawn, check, progress. No command for
the end of the lifecycle:

```
planned → [launch|spawn] → active → ... → ???
```

Closing out requires 9 manual steps per session (review PR, close
each issue, edit session.yaml, verify artifacts, reconcile concept
nodes, update engagements, cleanup worktree, validate, commit).

### 4.2 Interface

**CLI:**

```
keel session complete <session-id>
  [--skip-artifact-check]
  [--skip-worktree-cleanup]
  [--dry-run]
```

**Slash command:**

```
/pm-session-complete <session-id> [closing-note]
```

### 4.3 CLI behaviour

1. Verify status is `active` or `waiting_for_*`.
2. **Artifact check:** required completion artifacts present.
3. **Issue readiness:** each issue closeable (not already done).
4. Transition `session.yaml.status` → `completed`.
5. Update `engagements[]` with `ended_at` and `outcome`.
6. **Worktree cleanup** (unless skipped).
7. Print: issues to close, sessions unblocked, worktrees removed.

### 4.4 Slash command behaviour

Wraps CLI and adds PM-level actions:

1. Run `keel session complete <id> --skip-worktree-cleanup`.
2. If PR exists, run `/pm-session-review <id>` if not reviewed.
3. **Concept node reconciliation:** for each issue's Definition of
   Done, check `[[node-id]] content_hash updated if behaviour
   diverged`. Compare node descriptions against the PR diff. Update
   diverged nodes so downstream sessions read accurate specs. Run
   `keel refs reverse <node-id>` to log affected downstream
   sessions.
4. Close each issue → `done` with completion comment.
5. `keel validate --strict`.
6. Remove worktrees.
7. Commit: `complete: <session-id> (KUI-2, KUI-3, ...)`.
8. Report: issues closed, nodes updated (with downstream impact),
   sessions unblocked, recommended next launches.

### 4.5 Error cases

| Code | Trigger | Fix hint |
|---|---|---|
| `complete/not_active` | wrong status | Check session status |
| `complete/missing_artifacts` | required artifacts absent | Agent must write them first |
| `complete/issue_not_closeable` | issue can't transition to done | Check status_transitions |
| `complete/worktree_dirty` | uncommitted changes | Commit/stash or `--skip-worktree-cleanup` |
| `complete/pr_not_merged` | PR exists but unmerged | Merge first or `--force` |

### 4.6 Lifecycle

```
create → launch/spawn → monitor → review → reconcile nodes → complete
```

## 5. Feature D — spawn refinements (amends v0.6)

### 5.1 Context: Claude Code CLI audit

An audit of `claude` CLI v2.1.110 found flags the v0.6 spawn spec
doesn't use. v0.7 amends the spawn invocation to use the correct
flags and adds a per-session configuration schema.

### 5.2 Corrected spawn invocation

**v0.6 spec's §2.3 (current, incorrect):**

```bash
nohup claude -p "..." \
  --allowedTools "Bash,Read,Edit,Write,Glob,Grep" \
  --max-turns 200 \
  --output-format json \
  --append-system-prompt "session: <session-id>" \
  > <log-path> 2>&1 &
```

**v0.7 corrected invocation:**

```bash
nohup claude -p "$(cat <plan-path>)

You are the <agent> agent for session <session-id>.
Execute the plan. Stop at stop-and-ask points.
Open a PR titled '<type>(<session-slug>): <summary>' when done." \
  --name "<session-id>" \
  --effort max \
  --model opus \
  --fallback-model sonnet \
  --permission-mode bypassPermissions \
  --disallowedTools "Agent" \
  --max-turns 200 \
  --max-budget-usd 50 \
  --output-format stream-json \
  --append-system-prompt "keel session: <session-id>; project: <project-slug>
If context is getting heavy, use /compact to free space.
Do not use the Agent tool — you are a single-agent session.
Do not use /batch — the work is already decomposed in your plan." \
  > <log-path> 2>&1 &
```

### 5.3 Flag-by-flag rationale

| Flag | Why | v0.6 status |
|---|---|---|
| `--name <session-id>` | Aligns Claude session name with keel session. Enables `--resume <name>`. Visible in `claude sessions list`. | Not in v0.6 (used no naming) |
| `--effort max` | Autonomous agents do real implementation — no reason to throttle reasoning. | Not in v0.6 (tried system-prompt hack) |
| `--model opus` | Complex implementation work benefits from strongest model. | Not in v0.6 (used default) |
| `--fallback-model sonnet` | Resilience against overload. Agent doesn't stall; falls back gracefully. | Not in v0.6 |
| `--permission-mode bypassPermissions` | Autonomous agents can't prompt for permission. One flag replaces the `--allowedTools` list. | v0.6 used `--allowedTools` list |
| `--disallowedTools "Agent"` | Enforces single-agent constraint. Agents follow the plan; they don't re-decompose via subagents or `/batch`. Observed: significant execution quality degradation when agents spawn subagents (context loss, weaker synthesis, cost multiplication, invisible to PM monitor). See §6.1. | Not in v0.6 |
| `--max-turns 200` | Safety cap. Unchanged from v0.6. | In v0.6 |
| `--max-budget-usd 50` | Hard cost ceiling. Prevents runaway loops from burning budget. | Not in v0.6 |
| `--output-format stream-json` | Real-time event stream for the monitor. Each line is a parseable JSON event (tool calls, messages, errors, budget). | v0.6 used `json` (single blob at end) |
| `--append-system-prompt` | Session + project context. Unchanged in purpose. | In v0.6 |

### 5.4 Session resumption by name

Since we set `--name <session-id>`, resume is:

```bash
claude -p "Continue where you left off." \
  --resume "<session-id>" \
  --effort max \
  --output-format stream-json \
  >> <log-path> 2>&1 &
```

No PID tracking needed. `--resume` finds the session by name from
Claude Code's session persistence (`~/.claude/projects/`). The
`--fork-session` flag can create a new session ID while preserving
conversation history — useful for re-engagement after a failed
attempt where you want a clean engagement record.

This replaces the v0.6 spawn spec's §2.6 which used PID-based
resume (`kill -0 <pid>` to detect dead sessions). Name-based resume
is more robust — survives machine reboots, doesn't depend on the
spawning process staying alive.

### 5.5 Per-session spawn configuration

`session.yaml` gains an optional `spawn_config` block set by the PM
at scoping time:

```yaml
spawn_config:
  model: opus                   # default: opus
  fallback_model: sonnet        # default: sonnet
  effort: max                   # default: max
  max_turns: 200                # default: 200
  max_budget_usd: 50            # default: 50
  permission_mode: bypassPermissions  # default: bypassPermissions
```

`keel session spawn` reads these and maps them to CLI flags. The
session.yaml values override project-level defaults (which can live
in `project.yaml.spawn_defaults`). CLI flags override both.

**Use case:** a small scaffolding session gets `model: sonnet`,
`max_budget_usd: 10`. A complex multi-issue session gets
`model: opus`, `max_budget_usd: 100`, `max_turns: 400`.

### 5.6 Monitor integration

The `stream-json` output format is the key enabler for the monitor
(§2). Each line in the log is a JSON event:

```json
{"type":"assistant","message":{"content":"I'll start by..."},"turn":1}
{"type":"tool_use","tool":"Edit","input":{"file_path":"..."},"turn":2}
{"type":"tool_result","output":"File updated","turn":2}
{"type":"usage","total_tokens":15234,"cost_usd":0.42,"turn":2}
```

The monitor parses these to derive:
- Current turn count and budget usage
- Latest tool call (what the agent is doing right now)
- Whether the agent is stuck (no events for N minutes)
- Whether the agent hit an error (error events in the stream)

This is push-based monitoring without needing the agent messaging
MCP server — we get it for free from Claude Code's output format.

## 6. Feature E — Claude Code integration + single-agent constraint

### 6.1 The constraint: one agent per session

Keel sessions are deliberately scoped as single-agent work. The PM
decomposes the project into sessions; each session is small enough
for one agent to execute without further decomposition. This is a
hard constraint, not a suggestion.

**Why:** Testing shows significant degradation in execution quality
when agents spawn subagents. The parent agent loses context, the
synthesis step is weaker, costs multiply (each subagent starts a
cold prompt cache), and the PM's monitor can't see into nested
agent calls. `/batch` re-decomposes work the PM has already
decomposed — that's wasted effort that produces worse results.

**Enforcement:** The spawn invocation explicitly disables the Agent
tool:

```bash
--disallowedTools "Agent"
```

This prevents the executing agent from using Claude Code's Agent
tool to spawn subagents. The agent can still use all direct tools
(Bash, Read, Edit, Write, Glob, Grep) — it just can't delegate.

The corrected spawn invocation from §5.2 becomes:

```bash
nohup claude -p "$(cat <plan-path>) ..." \
  --name "<session-id>" \
  --effort max \
  --model opus \
  --fallback-model sonnet \
  --permission-mode bypassPermissions \
  --disallowedTools "Agent" \
  --max-turns 200 \
  --max-budget-usd 50 \
  --output-format stream-json \
  --append-system-prompt "keel session: <session-id>; project: <project-slug>" \
  > <log-path> 2>&1 &
```

### 6.2 Claude Code features the PM leverages

These are PM-level tools, not agent-level. The PM session (or the
human) uses them to manage executing sessions from outside:

| Feature | How the PM uses it | Who initiates |
|---|---|---|
| `/autofix-pr` | After an agent opens its PR, the PM (or monitor) attaches an autofix watcher. Handles CI failures without re-engaging the session agent. | PM or monitor, after PR opens |
| `/schedule` | Periodic project health checks as an alternative to `/loop` for long-running monitoring. | Human or PM |
| `/loop` | Self-paced monitoring loop (§2). | PM session |
| `/remote-control` | Make the PM session observable from claude.ai for human oversight. | Human |
| `/teleport` | Pull a web-based PM session into a local terminal for debugging. | Human |
| `/diff` | PM uses this during `/pm-session-review` to inspect changes. | PM |

### 6.3 Agent self-care instructions

The executing agent can't spawn subagents, but it should manage
its own context. Add these to the `--append-system-prompt`:

```
If your context is getting heavy, use /compact to free space.
Do not use the Agent tool — you are a single-agent session.
Do not use /batch — the work is already decomposed in your plan.
```

### 6.4 What /batch is for (not keel sessions)

`/batch` is valuable for ad-hoc work where there's no pre-scoped
plan: "migrate all files from X to Y", "add logging to every
endpoint." It decomposes on the fly and parallelises.

Keel sessions are the opposite: pre-decomposed by a PM agent who
has read the planning docs, written acceptance criteria, and set
dependency chains. The agent follows the plan. `/batch` would
ignore all of that context and re-decompose from scratch.

If a session feels too big for one agent, the answer is to split
it into two sessions at scoping time — not to let the agent
re-decompose at runtime.

### 6.5 spawn_config addition

```yaml
spawn_config:
  model: opus
  fallback_model: sonnet
  effort: max
  max_turns: 200
  max_budget_usd: 50
  permission_mode: bypassPermissions
  disallowed_tools:               # NEW
    - Agent                       # always — single-agent constraint
```

`disallowed_tools` defaults to `["Agent"]` and is not overridable
per-session in v0.7. A future version could allow it for specific
orchestration patterns, but the default is: agents don't spawn
agents.

## 7. Scope summary

**v0.7 (this spec):**

Monitor:
- `keel session monitor` CLI (one-shot snapshot)
- `/pm-session-monitor` slash command (self-paced loop)
- Primary: stream-json log tailing (real-time)
- Fallback: git/GitHub polling
- Auto-trigger `/pm-session-review` on PR open
- Alert on failure, stuck, cost threshold

Review:
- `keel session review` CLI + `/pm-session-review` slash command
- Per-issue acceptance criteria verification against PR diff
- Deviation detection (unspec'd files, deps, layout)
- Plan adherence checking
- Stop-and-ask audit

Complete:
- `keel session complete` CLI + `/pm-session-complete` slash command
- Concept node reconciliation (update diverged nodes, log downstream impact)
- Bulk issue close with completion comments
- Worktree cleanup
- Next-wave recommendations

Spawn refinements (amends v0.6):
- `--name`, `--effort`, `--model`, `--fallback-model` flags
- `--permission-mode bypassPermissions` replaces `--allowedTools`
- `--disallowedTools "Agent"` enforces single-agent constraint
- `--max-budget-usd` cost ceiling
- `--output-format stream-json` for monitor
- Resume by name instead of PID
- `spawn_config` schema in session.yaml

Claude Code integration:
- Single-agent constraint: agents cannot spawn subagents
- `/autofix-pr` attached by PM after agent opens PR
- Agent self-care instructions (`/compact`, no `/batch`)
- PM uses `/loop`, `/schedule`, `/remote-control` for oversight

**v0.8+ (deferred):**
- Agent messaging integration (push events beyond stream-json)
- Container-aware monitoring (container status, resource usage)
- Multi-project portfolio monitor (across workspace)
- Desktop/push notifications for critical events
- Auto-remediation (re-engage agent on CI failure without human)

## 7. Open questions

- **Should the monitor commit status updates to the PM repo?**
  Lean: yes, batched per tick — one commit per check, not per event.
- **Should the monitor have write access to the executing agent's
  worktree?** Lean: no — stay in PM role, create fix issues.
- **How does the monitor handle the human stepping in?** Lean:
  attribute all branch commits to the session regardless of author.
- **Should `spawn_config.permission_mode` ever be anything other
  than `bypassPermissions`?** For sessions touching sensitive code
  (security, payments), `acceptEdits` might be preferred — but
  then the agent prompts for permission with no one to answer.
  Lean: `bypassPermissions` for all autonomous sessions; human-
  supervised sessions use interactive `claude` (not `claude -p`).
- **`--exclude-dynamic-system-prompt-sections`**: improves prompt
  cache reuse across sessions sharing the same base prompt. Worth
  adding? Lean: yes, as a default — sessions already get their
  context via `--append-system-prompt`.
