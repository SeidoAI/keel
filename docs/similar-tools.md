# Similar tools — Keel in context

An analysis of eight adjacent tools and what Keel can learn from (or innovate over) each. Each section covers a summary, what to take, where Keel differs, and where Keel is innovative relative to the tool in question.

---

## Org-mode

**Summary.** Org-mode is a GNU Emacs major mode that turns plain-text `.org` files into a polymathic personal system: hierarchical outliner, TODO tracker, agenda view, tagger, clocking system, spreadsheet, literate-programming environment (via Babel), and note-capture interface — all in one tool. Any headline can carry a TODO/DONE state, deadline, schedule, tags, a property drawer with arbitrary key-value metadata, and embedded source blocks that execute in place. The agenda aggregates TODO items across multiple `.org` files into a single filterable calendar-like view.

It's designed for one human power user operating from Emacs, with data stored as plain text that plays well with git, Dropbox, or any sync tool. The design ethos — "make the easy things trivial and the complex things practical" — lets you start with a single scratch file and scale to a lifelong personal knowledge system with thousands of entries, custom agenda filters, and literate-code notebooks. There is no schema, no validation, no multi-user coordination model — just one brain imposing order on an outline.

**What to take.** The property drawer pattern (arbitrary `:key: value:` metadata attached to any headline) is a lightweight way to let users extend entities without schema changes — Keel's strict YAML schemas could grow a "drawer" for project-specific extensions that don't need to be part of the core validate gate. More concretely, Org's agenda view — aggregating TODOs from many files into one actionable feed — is directly analogous to something Keel should grow: a `keel agenda` command that reads every issue and session across the project and produces a single cross-cutting "what's in flight, owned by whom, due when" view. That's five lines of YAML parsing plus Rich rendering and would cost almost nothing.

**Where we differ.** Org-mode is single-user by design and assumes one brain is ordering the outline. There's no schema, no validation gate, no distinction between "issue" and "decision" and "endpoint" — any headline can be anything. Keel's use case assumes concurrent AI agents writing to a shared repo, which requires strong typing plus a drift-detection gate to prevent incoherent state. Org trusts the human to notice. Keel can't, because its user forgets every invocation.

**Where Keel is innovative vs. Org-mode.** Content-hashed concept nodes + the validate gate's drift detection. Org-mode has links between headlines, but they're just navigation; change the content of a headline and nothing notices that another headline's assumption about it is now stale. Keel's SHA-256 hashing on nodes + referrer tracking give coherence guarantees Org fundamentally doesn't attempt, because Org's user didn't need them.

## Fossil

**Summary.** Fossil is a distributed version control system by D. Richard Hipp (SQLite's author) that bundles VCS, bug tracking, wiki, forum, chat, technotes, a web UI, and role-based access control into one self-contained executable. A Fossil repository is a single `.fossil` file — actually a SQLite database — containing every artifact: code history, tickets, wiki pages, forum posts, and even web-UI state. Run `fossil ui` and you get GitHub-in-a-box locally. It's used by SQLite itself, tcl, and a small fiercely loyal community.

Philosophically it's the "cathedral" model to git's "bazaar." Fossil emphasizes durability (commits are permanent by design, history rewriting is deliberately hard), transparency (every project conversation lives inside the repo), and integration (tickets and wiki sync automatically with code because they *are* part of the repo). This contrasts sharply with git, where bug tracking, code review, and documentation have been outsourced to external SaaS — GitHub, Jira, Notion — that are neither versioned nor guaranteed to exist in five years.

**What to take.** Fossil is the closest philosophical ancestor to Keel — both bet that project artifacts belong *inside* the source repo, not in external SaaS. From Fossil we should borrow the "durability" framing explicitly in the README: tickets and decisions have long tails (you need to know why X was decided three years ago), and external trackers notoriously lose that history when a company migrates tools. Also practical: Fossil's unified timeline view (commits + tickets + wiki edits + forum posts interleaved chronologically) is something Keel could reproduce cheaply via `git log` + YAML parsing, and it would give humans a readable "what happened on this project" feed without needing a UI.

**Where we differ.** Fossil's target user is a small tightly-coordinated team of *humans* (SQLite itself has ~3 active committers). Its entire UI assumes a person clicking around a web interface, filing tickets through HTML forms, reading timeline pages. Keel's target user is Claude Code — an agent that doesn't benefit from a web UI and needs machine-queryable structure (YAML files with a strict schema, not SQL blobs behind a web form). Fossil also has no concept of "concept node" or drift detection; its tickets are free-form text, not graph-linked typed entities. And Fossil is a VCS, which Keel explicitly is *not* — Keel delegates versioning to git rather than reinventing it.

**Where Keel is innovative vs. Fossil.** Fossil puts artifacts in the repo but they're still semi-structured web content — a Fossil ticket is a form with text fields. Keel takes the next step: artifacts are *strictly-schemed YAML enforced by a single validate command*, which makes them programmable in a way Fossil's tickets aren't. Second, Keel ships the *methodology that produces the artifacts* (the PM skill, the slash commands) alongside the artifacts themselves. Fossil ships a web UI for humans; Keel ships a workflow skill for agents, versioned with the project. That's the generalization Fossil doesn't attempt: if you're going to put artifacts in the repo, put the procedure that produces them in the repo too.

## superpowers

**Summary.** Superpowers is a Claude Code plugin shipping a rigid workflow-discipline family of skills: `brainstorming` (required before any creative work), `writing-plans` (required before multi-step implementation, produces checkbox-granular plans with exact file paths and complete code), `executing-plans` (loads a plan and walks it task-by-task), `subagent-driven-development` (dispatches a fresh subagent per task with two-stage review), `test-driven-development` (hard iron-law TDD), `systematic-debugging`, `verification-before-completion`, and wrappers like `using-git-worktrees`, `finishing-a-development-branch`, `requesting/receiving-code-review`. The skills chain: `writing-plans` requires brainstorming first, `executing-plans` requires a written plan, `finishing-a-development-branch` runs after execution, and so on.

The stance is unusually rigid: "if you think there's even a 1% chance a skill might apply, you ABSOLUTELY MUST invoke it," "violating the letter of the rules is violating the spirit of the rules," "no exceptions." The authors clearly learned that agents rationalize their way out of discipline, so the skill text is defensive against those rationalizations with red-flag tables and iron-law framings. The plans themselves are structured as checkbox checklists with 2-5 minute bite-sized steps containing complete code, exact file paths, and expected command output — written for "a skilled developer who knows almost nothing about our toolset or problem domain."

**What to take.** Three things, in descending order of how much Keel already owes to superpowers. First: the bite-sized step structure of `writing-plans` is exactly what Keel's session-artifact plan files should look like — complete code, exact paths, 2-5 minute actions, no "TBD" or "handle errors." We should steal their plan document header format verbatim as Keel's plan artifact template. Second: their "required sub-skill" chaining (plans → execution → finishing) is a pattern Keel's `project-manager` skill should adopt — `/pm-scope` explicitly hands off to an implementation skill that explicitly hands off to verification, enforced by skill text, not left to the agent to remember. Third: their anti-rationalization red-flag tables (`| Thought | Reality |`) are a specific technique Keel's skill can borrow to pre-empt agent drift around validate-gate skipping.

**Where we differ.** Superpowers is a skill family for *any* coding task on *any* repo. It assumes a git worktree, a human partner in the loop, and an engineering activity (feature, bug, refactor). It does not assume a structured project dataset, a concept graph, or multi-agent coordination — it's a discipline layer for one agent doing one engineering task. Keel assumes the opposite: a structured corpus of hundreds of YAML artifacts, multiple agents coordinating via those artifacts, and PM work as a first-class activity distinct from coding. Superpowers would fit *inside* a Keel session as the skill a coder agent loads when executing an issue; it's a layer below Keel, not a peer.

**Where Keel is innovative vs. superpowers.** Superpowers is installed globally and applies to every project uniformly. It's opinionated about *how to work* but agnostic about *what your project looks like*. Keel inverts this: it ships the skill *inside the project repo*, versioned with the project. Different projects can carry different methodologies — a startup ships a lean PM skill, a regulated enterprise ships a compliance-heavy one, and an agent pulling the repo gets the right methodology automatically without global config. Superpowers can't do this because it lives in global config. Second, Keel has an actual persisted project dataset for the skill to operate on; superpowers has no concept of project state beyond "the code in the worktree." Keel is roughly what you'd build if you took superpowers' discipline seriously and asked: *what would the project itself need to look like to make the discipline cheap and automatic?*

## TDD family

**Summary.** TDD comes from Kent Beck's 2002 *Test-Driven Development: By Example*, out of the Extreme Programming community. The iron law: write a failing test, watch it fail (crucially — a test written after the code passes immediately and passing immediately proves nothing), write minimal code to pass, refactor while keeping tests green, repeat. The epistemic insight is that only a test you've seen fail for the *right reason* gives confidence it will catch future regressions. Everything else is theatre.

The family has branched widely: classical/"Detroit-style" (minimize mocks, let real objects collaborate) vs mockist/"London-style" (mocks to isolate units); BDD/Behavior-Driven Development (natural-language scenarios via Cucumber/Gherkin driving tests from stakeholder requirements); ATDD/Acceptance-Test-Driven Development (start with boundary tests, drive inward); property-based testing (QuickCheck, Hypothesis — generate hundreds of inputs and assert invariants). Second-generation thinkers like Martin Fowler, J. B. Rainsberger, and Gojko Adzic refined and critiqued. Beck himself has moved on to derivatives like test-commit-revert. The superpowers TDD skill above is a direct descendant of the rigid strain.

**What to take.** TDD's core insight — *you don't know if a test works until you've seen it fail* — generalizes directly to Keel's validate gate. The TDD analogue would be: every PM skill workflow produces a *failing validate* first (via a scaffold with deliberate gaps), then drives the workflow to make it green. Concretely: `keel scaffold-for-creation` could emit artifacts that are *deliberately incomplete* — missing required fields, unresolved `[[refs]]` — and the workflow is "fill in fields until `validate --strict` exits 0." That converts the PM skill from "write files and hope" into "scaffold → red → fill → green." Same epistemic crispness TDD gives code, applied to PM artifacts. This is the most actionable takeaway in this whole document.

**Where we differ.** TDD operates on code and treats the test suite as the spec. The red-green-refactor cycle doesn't cleanly map to "write an issue file," because there's no runtime behavior to test — just a document that conforms to a schema or doesn't. TDD also assumes one agent working on one unit at a time; Keel assumes multiple agents working on multiple issues concurrently, and the "test" (`validate --strict`) runs against the *whole project corpus* each time, not a single unit. The iron law "no code without a failing test" translates awkwardly to "no issue without a scaffold that starts invalid" — which is workable, but philosophically different.

**Where Keel is innovative vs. TDD.** TDD never had to solve "how do you detect that a decision made six months ago is now inconsistent with a new endpoint being added this week." Code tests don't have that problem because tests are pinned to current code — they fail immediately if current code changes. Keel's drift-detection-via-content-hash on concept nodes is the analogue for a domain TDD doesn't cover: the coherence of a slowly-evolving project corpus *across time*, not the correctness of code at one point in time. TDD protects against regression; Keel protects against drift. These are different failure modes.

## First interim observation (after the first four tools)

A pattern I noticed while comparing: each tradition owns a **coherence mechanism for a different unit**.

- **TDD**: coherence within a single code unit (test + code)
- **Superpowers**: coherence within a single engineering task (brainstorm → plan → execute → verify → finish)
- **Fossil**: coherence within a single codebase over time (code + tickets + wiki + timeline in one repo)
- **Org-mode**: coherence within a single human's attention (one outline, one agenda, one agent)
- **Keel**: coherence within a slowly-evolving multi-agent project corpus (concept graph + validate gate + methodology shipped in-repo)

Viewed this way, Keel isn't "another project tracker" — it's filling a gap in the coherence-mechanism ladder. The gap is: *how do you keep a project coherent when the actors are forgetful and the corpus grows over months?* None of the prior art addresses this specifically because none assumed a forgetful agent author. TDD assumes a developer with short-term memory. Superpowers assumes one coherent session. Fossil and Org-mode assume humans with long-term memory. Keel's innovation — if you accept the framing — is engineering a coherence mechanism calibrated to a forgetful multi-actor corpus.

The corollary worth surfacing: **Keel's value scales with how forgetful its users are.** For a human solo dev, git + a README is enough, because human memory closes the gap. For agents today, git alone leaves the corpus incoherent within weeks, and nothing short of a validate-gated concept graph keeps it stable. So "agent-first" isn't a marketing angle — it's a technical prerequisite, rooted in the current generation of agents having no cross-session episodic memory. Corollary of the corollary: if Claude eventually gets robust episodic memory across sessions, a meaningful chunk of Keel's coherence machinery becomes redundant. Worth tracking as a strategic risk — not because it invalidates Keel, but because it would shrink the gap Keel fills and shift the value toward the parts that remain even with perfect memory (the per-project methodology shipping, the strict schemas, the git-native durability). Those would survive; the drift-detection-as-external-memory part might not need to work as hard.

## dbt (data build tool)

**Summary.** dbt is a data-transformation framework where you write SQL `SELECT` statements as "models" in files, and dbt figures out the DAG, handles materialization, runs everything in dependency order, and gates the output on tests. A dbt project is a directory of `.sql` model files, `.yml` config/test files, `dbt_project.yml`, optional macros, seeds, snapshots, and source definitions. Users commit this directory to git and run `dbt build` to rebuild everything in order. dbt coined and popularized the term "analytics engineering" — applying software-engineering practices like version control, modularity, CI/CD, and testing to data workflows that were previously done in ad-hoc notebooks and one-off SQL scripts.

The two load-bearing pieces are the **lineage DAG** and the **`dbt build` gate**. dbt parses `ref()` and `source()` calls across all models to build a directed acyclic graph of who depends on whom, then uses that graph to (a) schedule builds in the right order, (b) skip downstream models if upstream tests fail, and (c) render an interactive lineage visualization in the generated docs site. Tests are either *generic* (schema-level: `not_null`, `unique`, `accepted_values`, `relationships` — the last enforces foreign-key-like referential integrity between model columns) or *singular* (arbitrary SQL queries that must return zero rows to pass). dbt-core is open source; dbt Cloud is the managed SaaS; the community is ~100k members and ships reusable test macros via packages like dbt-utils.

**What to take.** This is the tightest philosophical parallel of any tool in either round: dbt's lineage DAG + `dbt build` gate is structurally almost identical to Keel's concept graph + `keel validate --strict`, just for SQL models instead of YAML artifacts. Three specific things to steal. First, **selectors**: `dbt build --select marketing+` builds `marketing` and all its downstream dependents. Keel should grow `keel validate --select SEI-42+` to re-validate an issue and its downstream referrers, for fast feedback during edits. Second, **`ref()` as statically parseable**: dbt's DAG is built at compile time from static parsing of `ref()` calls. We already do this with `[[node-id]]`, but should expose it via `keel graph --upstream SEI-42` and `--downstream SEI-42` commands for explicit navigation. Third, the **`relationships` test** is dbt's most underrated feature — named, first-class referential-integrity assertions. Keel's validate gate should surface its referential checks as *named* test categories rather than one opaque "validation error" bucket.

**Where we differ.** dbt operates on data in a warehouse; Keel operates on project artifacts in a repo. dbt's "tests" run SQL queries against rows in a table and fail if rows come back — a runtime data check. Keel's "validation" is static schema + reference checking against YAML files — a structural check with no runtime. dbt's users are analytics engineers writing SQL; Keel's are agents writing YAML. And dbt's DAG is a homogeneous dependency graph with one kind of edge ("A references B"); Keel's concept graph has typed nodes (Decision, Contract, Endpoint, Model, Config) and cares about drift across time, which dbt doesn't because a model is rebuilt from scratch on every run.

**Where Keel is innovative vs. dbt.** dbt gates builds on tests but has no concept of *temporal coherence* — change a model and the old version simply ceases to exist. There's no "is the new version consistent with the decisions made six months ago?" question because there are no persistent decisions. Keel's content-hashing on concept nodes gives coherence across time for artifacts that outlive any single build — the thing dbt throws away on every run, Keel keeps and protects. Second: dbt ships a CLI and a package ecosystem but does *not* ship an instruction set telling you *how* to do analytics engineering; that lives in blogs, courses, and the dbt Labs "Coalesce" conference. Keel ships the methodology in-repo as the project-manager skill, versioned with the project. If dbt were more Keel-like, it would ship a "data-modeler" skill in-repo describing how to design a star schema, versioned alongside the models.

## Obsidian

**Summary.** Obsidian is a local-first personal knowledge base built on a directory of markdown files called a "vault." The defining feature is the `[[Note Name]]` wikilink syntax: type `[[`, pick a target, and Obsidian creates a bidirectional link that renders as a backlink on the destination. The graph view visualizes the whole vault as a force-directed network of nodes (notes) and edges (links), letting users see the shape of their knowledge. Obsidian also has tags, frontmatter properties (YAML at the top of a file for structured metadata), and a plugin ecosystem that rivals VS Code's in breadth — community plugins for everything from Kanban boards to spaced repetition to task management.

Philosophically, Obsidian is militantly local-first: your vault is just a folder, files are just markdown, and the app never phones home. You can sync via Obsidian's paid Sync service, git, Dropbox, iCloud, or a USB stick — Obsidian doesn't care. The target user is a single human doing Zettelkasten-style personal knowledge management, digital gardening, or "second-brain" building. The differentiator from Notion is "your data on your disk, forever, in a format you can read with `cat`" — no vendor lock-in, no server dependency, no block-level proprietary format. The differentiator from Roam Research is being local-first instead of SaaS-first, plus a dramatically larger plugin community.

**What to take.** Obsidian is the popular ancestor of Keel's `[[node-id]]` syntax (even if we arrived via ADR tools or wiki conventions). Two concrete things to borrow. First, **backlinks as a first-class surface**: Obsidian shows "mentioned in these other notes" at the bottom of every note, which is how users actually navigate. Keel should render backlinks for every concept node — "this decision is referenced by SEI-12, SEI-34, SEI-41, dec-auth-rotation" — via `keel node show <id>` or a dedicated `keel refs <id>` command. We have the data; we just need to surface it. Second, **a JSON export of the graph for agent working memory**: humans browse Obsidian's graph visually, but agents can't parse SVG efficiently. Emitting `keel graph --format=json` as a clean machine-readable dump of all nodes and edges would turn the concept graph from a coherence checker into an agent's external working-memory index: "what's connected to the auth endpoint?" becomes one command away.

**Where we differ.** Obsidian is single-user and schema-less by design. Any `[[link]]` can point to any note, notes can contain anything, typed entities don't exist, and there are no required fields. That's a feature for a human working on loose ideas — but a bug for an agent trying to keep a shared project coherent. Keel has typed node kinds, validated schemas, a drift-detection gate, and mandatory fields per type. Obsidian trusts the human author to know what they meant; Keel can't and enforces structure instead. Also, Obsidian has no orchestration model — it's a thinking tool, not a doing tool. There is no "session" in Obsidian, no executor, no verification-checklist ceremony.

**Where Keel is innovative vs. Obsidian.** Obsidian's graph is *descriptive* — it shows you what's linked. Keel's graph is *prescriptive* — it enforces that links resolve, rejects broken refs at the validate gate, and flags content drift when a referent changes. Obsidian will happily let you create a broken `[[doesnt-exist]]` link and render it in red; Keel refuses to accept that state into main. Second, Obsidian is purely a knowledge tool — you can't "execute" a note. Keel's concept nodes are tied to an orchestration model (sessions, issues, executors) where agents *act* based on what the graph says. Obsidian is a thinking tool that can't do; Keel is a doing tool with coherent thinking built in.

## Linear

**Summary.** Linear is a hosted issue-tracking and project-management tool for software teams, built as the opinionated, fast, keyboard-first alternative to Jira. Its data model has Teams, Projects (multi-issue initiatives with goals), Cycles (time-boxed sprints, typically 1-2 weeks), Issues (the atomic work unit with state, priority, assignee, estimate, labels, parent/child, blockers), Workflows (custom state machines per team: Backlog → Todo → In Progress → In Review → Done → Canceled), Milestones (project checkpoints), and Roadmaps (strategic rollup). Every keystroke is a shortcut, every view is filterable, every change syncs in real time across every connected client. It's the de-facto choice for Series A-C startups over the last five years, displacing Jira for modern product-engineering teams.

Linear is *opinionated where Jira is configurable*. It ships a default workflow that works for ~80% of software teams out of the box, a fast React app, a clean GraphQL API, first-class GitHub/GitLab integrations, Slack notifications, and a mobile app. The company (Linear Inc.) is a SaaS business with paid plans; your data lives in their Postgres, accessed through their app or their API. The target user is a software engineering team that wants to plan and ship without the overhead of a Jira admin. This is the tool Keel is literally positioned to replace for the AI-agent use case — not because Linear is bad, but because Linear's mental model assumes a human driver, and we're betting the driver is changing.

**What to take.** Two things. First, **the opinionated default workflow**. Linear's bet is that most teams don't benefit from configuring custom workflows — they benefit from a *good default* that's hard to misuse. Keel should ship a good default workflow state enum (todo / in_progress / in_review / done / canceled / blocked), make customizing it possible but not encouraged, and surface that default prominently in `keel brief` output and the PM skill. We have the enum templates; we should make them more prominent. Second, **keyboard-first for humans translates to "CLI-first with zero interactivity" for agents**. Linear built for humans flying through the app without touching a mouse; the agent analogue is "one command does one thing with predictable flags, no interactive prompts, no fuzzy matching." Keel already does this, but should aggressively hold the line against future interactive features. The only exception should remain `keel init` for first-time humans.

**Where we differ.** Linear is a hosted SaaS where data lives in multi-tenant Postgres, state changes go through a real-time sync layer, and the UX is optimized for humans clicking and typing in rich-text editors. Keel is a CLI + a repo where data lives in YAML files, state changes *are* git commits, and the "UX" is a skill that agents load. Linear optimizes for human collaboration speed (fast UI, real-time updates, notifications, keyboard shortcuts); Keel optimizes for agent correctness (strict schemas, drift detection, validate gate, methodology in-repo). The deeper difference: Linear's bet is that the bottleneck for PM work is the humans' UI speed. Keel's bet is that the bottleneck has moved — agents are doing more PM work now, and the bottleneck for agents is *coherence*, not UI speed.

**Where Keel is innovative vs. Linear.** Three things. First, **versioned methodology**: Linear's workflow is configured per-team in Linear's UI; if you change jobs and want to re-use the same methodology, you rebuild it from scratch in a new Linear workspace. Keel ships the methodology in the repo — fork the repo, get the methodology for free, including the PM skill, the slash commands, and the workflow enums. Second, **durable history**: Linear issues live in Linear's Postgres; if Linear the company disappears, your issues might too (they have exports, but you'd rebuild the tooling). Keel issues are git commits — they survive any SaaS death and are readable with `cat` in 20 years. Third, **agent-first mechanisms**: Linear has a good API, which is "agent-adjacent" but not agent-first — an agent using Linear's API still has to invent its own methodology, remember state between API calls, and work around the fact that Linear's data model assumes a human is driving. Keel's data lives where the agent reads code, and the methodology is shipped alongside the data.

**Honest caveat.** Linear's UI is genuinely better than anything Keel will ever ship, because Keel is deliberately not trying to ship a UI. If a human wants to browse their work on a phone at 11pm, or visually drag issues across cycles, or leave a rich-text comment with a @mention, Linear wins decisively. The Keel bet is "when agents do most of the work, the human doesn't need the UI as often." That bet is unproven; if it's wrong, Keel's value shrinks significantly for human-heavy teams. Worth being clear-eyed about this, especially when pitching.

## Terraform

**Summary.** Terraform is HashiCorp's infrastructure-as-code tool where you declaratively describe cloud resources in HCL (a configuration language with expressions, loops, conditionals, and variables), and Terraform figures out how to create, update, or destroy them to match. The core pieces: **providers** (plugins that speak the API of AWS/GCP/Azure/Kubernetes/etc. — thousands exist), **resources** (individual pieces of infrastructure: an EC2 instance, a DNS record, a Postgres database), **modules** (reusable bundles of resources with inputs and outputs, published via the Terraform Registry), **state** (a JSON file mapping HCL declarations to actual resource IDs in the cloud), and **HCL** itself.

The lifecycle is `init` → `plan` → `apply`. **`plan`** is the signature feature: Terraform reads your HCL config, reads the current state file, optionally refreshes against reality, and prints a human-readable diff showing exactly what will be created, updated, or destroyed. **`apply`** executes the plan in dependency order, parallelizing independent resources, updating state atomically. **"Drift detection"** is Terraform's term of art for "reality has diverged from state" — someone clicked in the AWS console, a resource was deleted out-of-band, a cron job mutated the DB. Running `terraform plan` surfaces both config-diff (what you want to change) and drift (what's already different from state). The philosophy is declarative: you describe the end state, Terraform computes the steps. Terraform is the dominant tool for managing multi-cloud infrastructure.

**What to take.** Four concrete things, in priority order. First, **the plan/apply dichotomy**: Terraform separates "show me what would change" from "execute the change," and that separation is load-bearing for trust. Keel should add `keel plan` that shows exactly what a PM workflow would write (issues to create, nodes to add, edges to form) before any file is written. Agents get a "look before you leap" gate currently missing. Second, **explicit drift detection as a named failure category**: Terraform's plan output distinguishes "drift" (reality ≠ state) from "change" (intended diff). Keel's validate gate conflates "something's wrong" into one bucket — we should distinguish *schema invalid* from *content-hash drift* from *unresolved reference* as three named kinds of failure in the validate report, not one opaque blob. Third, **modules as versioned reusable patterns**: the Terraform Registry lets teams publish `hashicorp/vault/aws` v3.21.0 and consume it by reference. Keel could grow a "skill registry" where project-manager skills for different domains (lean startup, enterprise compliance, academic research) are versioned and installable into new projects. Fourth, **`terraform refresh`**: updates state from reality without changing anything. Keel's analogue is `graph_cache.ensure_fresh`, which we have — but we should expose it as a named `keel refresh` command rather than hiding it inside validate.

**Where we differ.** Terraform manages cloud infrastructure with a massive blast radius (a typo can delete a production database); Keel manages project artifacts in a repo with almost no blast radius (worst case: broken validate, fixed with `git reset`). Terraform's state file is a critical single source of truth that must be locked (via S3 + DynamoDB or Terraform Cloud) for safe concurrent access; Keel's state is git, with concurrent access resolved by merge. Terraform's "providers" are external APIs that fail, rate-limit, and diverge from state; Keel's "providers" are local filesystem reads that are deterministic and cheap. And Terraform's users are DevOps/platform engineers making careful one-at-a-time changes; Keel's users are agents making many cheap changes validated in bulk.

**Where Keel is innovative vs. Terraform.** Terraform pioneered drift detection as a concept, but *Terraform drift is about reality diverging from declared state* — the state file remains the source of truth. Keel's drift is about *project artifacts diverging from each other* — there is no single source of truth, there's a corpus of mutually-referential documents that must stay coherent with each other. Different problem, same word. Second, Terraform's plan/apply assumes one actor at a time (hence state locking); Keel assumes multiple agents reading and writing concurrently via git merge, with the validate gate as the reconciler. Keel is designed for coordination-free concurrency; Terraform is designed for mutually-exclusive access. Finally: Terraform ships no methodology. You learn HCL syntax and then you're on your own to figure out how to organize a real project. (The community has produced "Terragrunt" and various opinionated wrappers, but HashiCorp itself doesn't ship one.) Keel ships the methodology in the repo.

---

## Synthesis — the pattern across all eight

The meta-pattern across all eight is that each owns a different **coupling between state, coherence, and methodology**:

| Tool | Methodology lives | State lives | Coherence mechanism |
|---|---|---|---|
| TDD | Developer's head | Code | Test suite passing |
| Superpowers | Global Claude config | Worktree | `finishing-a-branch` skill |
| Fossil | Implicit in conventions | One SQLite file | Versioned tickets in same repo |
| Org-mode | Personal | `.org` files | User's memory |
| dbt | External practice (blogs, courses) | Models + warehouse | `dbt build` + tests |
| Obsidian | Personal (Zettelkasten etc.) | Vault of markdown | User's intent |
| Linear | Baked into the UI | Postgres | Workflow state machine |
| Terraform | Not shipped | State file | Plan/apply lifecycle |
| **Keel** | **In-repo as a skill** | **Git + YAML** | **Validate gate + concept graph + drift detection** |

Keel is the only one that *ships the methodology alongside the state, versioned together*. dbt comes closest but treats analytics engineering as an external practice, not a versioned artifact. Everything else either externalizes methodology (Terraform, Jira) or leaves it implicit (Fossil, Obsidian, Org-mode). If there's a single distinctive axis to Keel, this is it — and it's not marketing, it's the gap the competitive landscape actually leaves open.

A second observation worth surfacing because it's uncomfortable: **in five of these eight tools (Linear, Fossil, Obsidian, Org-mode, and to a lesser extent dbt Cloud), the primary user experience is visual.** Humans navigate UIs, hover on graph nodes, click through cycles, scan agendas. Keel deliberately has no UI and maps very poorly onto human visual navigation. If a senior manager walks into your office and asks "show me the state of the project," Keel's current answer is "`cat issues/ && cat sessions/`." That's a genuine weakness and we shouldn't romanticize it away. A minimal read-only HTML viewer — one command, `keel view`, localhost only, no backend, renders the concept graph + open issues + active sessions into a skimmable page — would dramatically lower the cost of selling Keel to a team that currently has a Linear-shaped mental model. Crucially, this wouldn't contradict the agent-first bet. It would be the human-in-the-loop escape hatch: agents do the work, the repo is the source of truth, and when a human needs to verify or brief in, they get a 30-second visual instead of having to read YAML. The humans in the loop still need a way to *trust* the state without reading files by hand. That's a future-session conversation, but it's the most concrete thing I'd add to the v1 roadmap based on this comparative analysis.
