# Schema: Issues

Issues live at `issues/<KEY>/issue.yaml`. Each is a YAML frontmatter +
Markdown body file. The canonical example is
`examples/issue-fully-formed.yaml` ‚Äî **if this doc disagrees with that
example, trust the example**.

## Directory layout

Each issue is a directory:

```
issues/<KEY>/
‚îú‚îÄ‚îÄ issue.yaml         # the issue definition (this schema)
‚îú‚îÄ‚îÄ comments/          # structured comments written during execution
‚îÇ   ‚îî‚îÄ‚îÄ 001-status-2026-04-07.yaml
‚îú‚îÄ‚îÄ developer.md       # completion notes from the dev agent
‚îî‚îÄ‚îÄ verified.md        # verification report from the verification agent
```

The directory name must match the issue's `id` field. `comments/`,
`developer.md`, and `verified.md` are produced during execution; the PM
agent only writes `issue.yaml` during scoping.

## Frontmatter fields

| Field | Type | Required | Notes |
|---|---|---|---|
| `uuid` | UUID4 | yes | Canonical identity. Generated by the agent (not a CLI). Never changes. |
| `id` | string | yes | Human-readable key in `<PREFIX>-<N>` form. Allocated via `tripwire next-key --type issue`. |
| `title` | string | yes | One-line summary. |
| `status` | string | yes | Must be a value in `templates/enums/issue_status.yaml`. |
| `priority` | string | yes | Must be in `templates/enums/priority.yaml`. |
| `executor` | string | yes | Must be in `templates/enums/executor.yaml` (`ai`, `human`, `mixed`). |
| `verifier` | string | yes | Must be in `templates/enums/verifier.yaml` (`required`, `optional`, `none`). |
| `agent` | string | no | Agent definition id (matches a file in `agents/`). |
| `labels` | list[string] | no | Free-form tags. |
| `parent` | string or null | no | Parent epic's key. |
| `repo` | string | no | GitHub slug; must be declared in `project.yaml.repos`. |
| `base_branch` | string | no | Defaults to project's `base_branch`. |
| `implements` | list[string] | no | Requirements or decisions this implements (e.g. `REQ-AUTH-001`). |
| `blocked_by` | list[string] | no | Other issue keys that must land first. |
| `blocks` | list[string] | no | **Do NOT write.** The validator computes this from inverse `blocked_by`. |
| `docs` | list[string] or null | no | Extra doc paths mounted into the container. |
| `created_at` | ISO datetime | yes | When the issue was created. |
| `updated_at` | ISO datetime | yes | When the issue was last touched. |
| `created_by` | string | no | Author handle. |

## Required body sections ‚Äî concrete issues

The validator checks that these Markdown headings are present for
concrete issues (any issue WITHOUT the `type/epic` label):

1. `## Context` ‚Äî what this issue is about, why it's needed, which
   concept nodes are involved (reference via `[[node-id]]`).
2. `## Implements` ‚Äî requirements or decisions being fulfilled.
3. `## Repo scope` ‚Äî target repo, base branch, primary paths, required config.
4. `## Requirements` ‚Äî bullet list, specific and actionable.
5. `## Execution constraints` ‚Äî must include "stop and ask" guidance.
6. `## Acceptance criteria` ‚Äî checkbox items (`- [ ] ...`), at least one.
7. `## Test plan` ‚Äî commands to run to verify.
8. `## Dependencies` ‚Äî other issues and concept nodes needed first.
9. `## Definition of Done` ‚Äî the final checklist.

If any heading is missing, the validator emits `body/missing_heading`.

## Required body sections ‚Äî epic issues

Epic issues (issues with `labels: [type/epic]`) have relaxed
requirements. The validator checks only these headings:

1. `## Context` ‚Äî what this epic covers and why it exists.
2. `## Child issues` ‚Äî table or list of child issue keys.
3. `## Acceptance criteria` ‚Äî checkbox items, at least one.

Epics do NOT require: Implements, Repo scope, Execution constraints,
Test plan, Dependencies, or Definition of Done. These belong on the
child issues, not the grouping container.

Epics are also NOT checked for "stop and ask" guidance (they are not
executed by agents). Node references (`[[node-id]]`) are optional
for epics (warning, not error).

See `examples/issue-epic.yaml` for the canonical epic format.

## Other body rules

- **At least one `[[reference]]`** ‚Äî if an issue has zero concept node
  references, the validator warns (`body/no_references`). This is a
  strong signal of a coherence gap. For epics this is a warning only.
- **"Stop and ask" guidance** ‚Äî concrete issues must contain the phrase
  "stop and ask" (or "stop, ask") somewhere. Not required for epics.
- **At least one acceptance checkbox** ‚Äî under `## Acceptance criteria`,
  there must be at least one `- [ ]` or `- [x]` item. Applies to both
  concrete issues and epics.

## File path

`<project>/issues/<KEY>/issue.yaml`. The filename (minus `.yaml`) must exactly
match the `id` field in frontmatter ‚Äî the validator checks this via the
ID collision detector.
