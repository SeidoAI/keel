# Schema: Issues

Issues live at `issues/<KEY>.yaml`. Each is a YAML frontmatter + Markdown
body file. The canonical example is `examples/issue-fully-formed.yaml` —
**if this doc disagrees with that example, trust the example**.

## Frontmatter fields

| Field | Type | Required | Notes |
|---|---|---|---|
| `uuid` | UUID4 | yes | Canonical identity. Generated by the agent (not a CLI). Never changes. |
| `id` | string | yes | Human-readable key in `<PREFIX>-<N>` form. Allocated via `keel next-key --type issue`. |
| `title` | string | yes | One-line summary. |
| `status` | string | yes | Must be a value in `enums/issue_status.yaml`. |
| `priority` | string | yes | Must be in `enums/priority.yaml`. |
| `executor` | string | yes | Must be in `enums/executor.yaml` (`ai`, `human`, `mixed`). |
| `verifier` | string | yes | Must be in `enums/verifier.yaml` (`required`, `optional`, `none`). |
| `agent` | string | no | Agent definition id (matches a file in `agents/`). |
| `labels` | list[string] | no | Free-form tags. |
| `parent` | string or null | no | Parent epic's key. |
| `repo` | string | no | GitHub slug; must be declared in `project.yaml.repos`. |
| `base_branch` | string | no | Defaults to project's `base_branch`. |
| `implements` | list[string] | no | Requirements or decisions this implements (e.g. `REQ-AUTH-001`). |
| `blocked_by` | list[string] | no | Other issue keys that must land first. |
| `blocks` | list[string] | no | **Do NOT write.** The validator computes this from inverse `blocked_by`. |
| `docs` | list[string] or null | no | Extra doc paths mounted into the container. |
| `created_at` | ISO datetime | yes | When the issue was created. |
| `updated_at` | ISO datetime | yes | When the issue was last touched. |
| `created_by` | string | no | Author handle. |

## Required body sections

The validator checks that these Markdown headings are present:

1. `## Context` — what this issue is about, why it's needed, which
   concept nodes are involved (reference via `[[node-id]]`).
2. `## Implements` — requirements or decisions being fulfilled.
3. `## Repo scope` — target repo, base branch, primary paths, required config.
4. `## Requirements` — bullet list, specific and actionable.
5. `## Execution constraints` — must include "stop and ask" guidance.
6. `## Acceptance criteria` — checkbox items (`- [ ] ...`), at least one.
7. `## Test plan` — commands to run to verify.
8. `## Dependencies` — other issues and concept nodes needed first.
9. `## Definition of Done` — the final checklist.

If any heading is missing, the validator emits `body/missing_heading`.

## Other body rules

- **At least one `[[reference]]`** — if an issue has zero concept node
  references, the validator warns (`body/no_references`). This is a
  strong signal of a coherence gap.
- **"Stop and ask" guidance** — the body must contain the phrase "stop
  and ask" (or "stop, ask") somewhere. The validator checks for this.
- **At least one acceptance checkbox** — under `## Acceptance criteria`,
  there must be at least one `- [ ]` or `- [x]` item.

## File path

`<project>/issues/<KEY>.yaml`. The filename (minus `.yaml`) must exactly
match the `id` field in frontmatter — the validator checks this via the
ID collision detector.
